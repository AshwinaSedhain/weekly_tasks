
# 1. WHAT IS THIS PROJECT?

This project is a Real-Time Data Pipeline that simulates how large-scale
platforms like Amazon, Flipkart, Netflix, or Swiggy track and analyze user
behavior on their platform  both in real time and historically.

Every time a user performs an action on a platform (logging in, clicking a
product, making a purchase), that action is called a "user activity event."
In real systems, millions of such events are generated every second. The
challenge is:

  - How do you CAPTURE all these events without losing any?
  - How do you PROCESS them in real time to get instant insights?
  - How do you ANALYZE the full historical data for business reports?

This project answers all three questions by building a complete end-to-end
data pipeline using Apache Kafka and Apache Spark — two of the most widely
used Big Data tools in the industry.

# 2. WHY WAS THIS PROJECT BUILT? (THE BUSINESS PROBLEM)


Imagine you are running an e-commerce platform. Your business team asks:

  REAL-TIME QUESTIONS (need answers within seconds):
  - Which users are currently active on the platform?
  - How many purchases happened in the last 10 seconds?
  - Is there a sudden spike in login failures right now?

  HISTORICAL QUESTIONS (need answers from all past data):
  - Which user made the most purchases this month?
  - What is the most visited page on our platform?
  - How many total events did each user generate?

A single system cannot efficiently answer both types of questions.
Real-time systems are fast but don't store everything.
Batch systems are thorough but slow.

The industry solution is called LAMBDA ARCHITECTURE — which runs both a
real-time layer and a batch layer in parallel. This project implements a
simplified but fully functional version of Lambda Architecture.


# 3. WHAT IS LAMBDA ARCHITECTURE?

Lambda Architecture is a data processing design pattern that handles both
real-time and batch data processing simultaneously.

It has three layers:

  INGESTION LAYER  →  Captures all incoming data
  SPEED LAYER      →  Processes data in real time (low latency)
  BATCH LAYER      →  Processes all historical data (high accuracy)
  SERVING LAYER    →  Combines and presents the final output

In our project:

  Apache Kafka        = Ingestion Layer
  Spark Streaming     = Speed Layer
  Spark Batch Job     = Batch Layer
  Console + CSV Files = Serving Layer

This is exactly how companies like LinkedIn (who invented Kafka), Uber,
and Netflix process their data at scale.

# 4. TECH STACK — TOOLS USED AND WHY

APACHE KAFKA:
  - Kafka is a distributed event streaming platform.
  - It works like a high-speed message queue  producers send messages,
    consumers read them.
  - We use Kafka as the central data bus. All user events flow through it.
  - Why Kafka? It is fault-tolerant, scalable, and can handle millions of
    events per second. It also stores events for a configurable period,
    allowing both real-time and batch consumers to read the same data.
  - In our project: Kafka holds the topic "user_events" with 3 partitions.
    3 partitions means 3 parallel streams of data, allowing faster processing.

APACHE SPARK (Structured Streaming + Batch):
  - Spark is a unified analytics engine for large-scale data processing.
  - Spark Structured Streaming is Spark's real-time processing engine.
    It treats a live data stream as an unbounded table and runs continuous
    SQL-like queries on it.
  - Spark Batch reads data as a static dataset and runs full aggregations.
  - Why Spark? It is the industry standard for both streaming and batch
    processing. It runs in-memory, making it extremely fast.

PYTHON:
  - Used to write the Kafka producer and both Spark jobs (PySpark).
  - Python is the most common language for data engineering pipelines.

JSON:
  - All events are serialized as JSON — lightweight, human-readable,
    and universally supported.

# 5. PROJECT ARCHITECTURE — HOW DATA FLOWS

STEP 1: EVENT GENERATION (producer.py)
  - producer.py runs continuously and generates fake user activity events.
  - Each event looks like this:
      {
        "user_id": "user_007",
        "event_type": "purchase",
        "timestamp": "2026-04-29T08:01:44.007920+00:00",
        "session_id": "sess_7924",
        "page": "/checkout"
      }
  - Events are sent to Kafka topic "user_events" at ~1 event per second.
  - 20 simulated users (user_001 to user_020) generate events with weighted
    probability: click=50%, login=30%, purchase=20% — mimicking real behavior.
  - The producer uses key=user_id so all events from the same user go to
    the same Kafka partition (ensures ordering per user).

STEP 2: KAFKA INGESTION (Ingestion Layer)
  - Kafka receives all events and stores them in topic "user_events".
  - The topic has 3 partitions — events are distributed across partitions
    based on the user_id key.
  - Kafka acts as a buffer between the producer and consumers. Even if
    Spark is temporarily down, no events are lost — they stay in Kafka.
  - Both the streaming job and the batch job read from the same Kafka topic
    independently, without interfering with each other.
    This is possible because each consumer maintains its own offset pointer.

STEP 3: SPARK STREAMING  SPEED LAYER (spark_streaming.py)
  - Spark Structured Streaming reads events from Kafka in real time.
  - It parses the JSON payload and performs two real-time operations:

  OPERATION A — 10-Second Tumbling Window Aggregation:
    - Groups events into 10-second time windows.
    - Counts how many events each user generated per event type
      within each window.
    - Uses watermarking (10 seconds) to handle late-arriving events.
    - Output example:
        +------------------------------------------+--------+----------+-----------+
        |window                                    |user_id |event_type|event_count|
        +------------------------------------------+--------+----------+-----------+
        |{2026-04-29 08:01:00, 2026-04-29 08:01:10}|user_009|purchase  |1          |
        |{2026-04-29 08:01:00, 2026-04-29 08:01:10}|user_003|click     |2          |
        +------------------------------------------+--------+----------+-----------+

  OPERATION B — Real-Time Purchase Filter:
    - Filters only "purchase" events from the stream.
    - Displays them instantly as they arrive.
    - This simulates a real-time fraud detection or sales monitoring feed.
    - Output example:
        +--------+----------+--------------------------------+----------+---------+
        |user_id |event_type|timestamp                       |session_id|page     |
        +--------+----------+--------------------------------+----------+---------+
        |user_012|purchase  |2026-04-29T02:16:44.007920+00:00|sess_7924 |/checkout|
        +--------+----------+--------------------------------+----------+---------+

  - Both queries run simultaneously as independent streaming queries.
  - Results refresh every 10 seconds automatically.

STEP 4: SPARK BATCH JOB... BATCH LAYER (spark_batch.py)
  - The batch job reads ALL historical events from Kafka at once
    (from the very first event to the latest).
  - It uses spark.read (not spark.readStream) — treating Kafka as a
    static data source, not a live stream.
  - It performs three full historical aggregations:

  AGGREGATION 1 ... User Activity Summary:
    - Total events per user
    - Total purchases per user
    - Total logins per user
    - Total clicks per user
    - Sorted by most active user
    - Result from our run (2,712 events):
        user_020: 159 events, 40 purchases, 43 logins, 76 clicks
        user_004: 154 events, 23 purchases, 54 logins, 77 clicks
        ... (all 20 users shown)

  AGGREGATION 2.... Event Type Breakdown:
    - Total count of each event type across all users
    - Result: click=1389, login=788, purchase=535
    - Confirms our weighted distribution is working correctly

  AGGREGATION 3....Top Pages:
    - Most visited pages across all users and sessions
    - Result: /product=560, /cart=551, /checkout=538, /home=536, /profile=529

STEP 5: SERVING LAYER (Output)
  - All batch results are printed to console in formatted tables.
  - All three aggregations are also saved as CSV files:
      output/batch_results/user_summary/
      output/batch_results/event_breakdown/
      output/batch_results/top_pages/
  - These CSV files can be loaded into any BI tool (Tableau, Power BI,
    Excel) for visualization and reporting.

6. WHAT EACH FILE DOES

producer.py
  - Simulates a real application generating user events.
  - Connects to Kafka and sends JSON events continuously.
  - Uses kafka-python-ng library (Python 3.12 compatible fork).
  - Configures acks=all for guaranteed delivery.

spark_streaming.py
  - The Speed Layer of the Lambda Architecture.
  - Uses PySpark Structured Streaming to read from Kafka in real time.
  - Runs two parallel streaming queries (windowed counts + purchase filter).
  - Displays results to console every 10 seconds.

spark_batch.py
  - The Batch Layer of the Lambda Architecture.
  - Uses PySpark batch mode to read all Kafka events at once.
  - Computes three serving layer aggregations.
  - Saves results as CSV files and prints to console.

requirements.txt
  - Python dependencies: kafka-python-ng==2.2.2, pyspark==3.5.0

run_tmux.sh / run_pipeline.sh
  - Automation scripts to start all components in the correct order.

output/batch_results/
  - Final serving layer output — CSV files with aggregated results.

8. KEY CONCEPTS DEMONSTRATED

KAFKA PARTITIONS:
  We created the topic with 3 partitions. This means Kafka splits the data
  into 3 parallel streams. Spark automatically assigns one task per partition,
  enabling parallel processing. With 3 partitions, 3 Spark tasks read data
  simultaneously — this is how Kafka achieves horizontal scalability.

CONSUMER GROUPS:
  Spark Streaming and Spark Batch both read from the same Kafka topic but
  maintain separate offset pointers. This means both can read all the data
  independently without one affecting the other. In production, you could
  have 10 different applications all reading the same Kafka topic
  simultaneously — each gets a full copy of the data.

WATERMARKING:
  In streaming, events can arrive late due to network delays. Watermarking
  tells Spark: "wait up to 10 seconds for late events before finalizing a
  window." This prevents incorrect counts due to out-of-order data.

STRUCTURED STREAMING:
  Unlike older DStream-based streaming, Structured Streaming treats the
  stream as an infinite DataFrame. This means all standard Spark SQL
  operations (filter, groupBy, join, window) work exactly the same on
  streaming data as on batch data — a major advantage for developers.

BATCH READING FROM KAFKA:
  Kafka stores all events for a configurable retention period (default 7 days).
  Spark can read Kafka in batch mode using startingOffsets=earliest and
  endingOffsets=latest — effectively treating the entire Kafka topic as a
  database table. This is the key insight that makes our simplified Lambda
  Architecture work without a separate data lake or HDFS.


9. SUMMARY


built a complete, working, end-to-end Big Data streaming pipeline that:

  1. GENERATES simulated user activity events (producer.py)
  2. INGESTS them into Apache Kafka (user_events topic, 3 partitions)
  3. PROCESSES them in real time using Spark Structured Streaming
     - 10-second windowed event counts per user
     - Real-time purchase event monitoring
  4. ANALYZES full historical data using Spark Batch
     - Total events, purchases, logins, clicks per user
     - Event type distribution
     - Top pages by visit count
  5. SERVES results via console output and CSV files

This implements a simplified Lambda Architecture — the same architectural
pattern used by LinkedIn, Netflix, Uber, and most large-scale data platforms
in production today.
